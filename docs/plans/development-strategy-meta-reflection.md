# harness-lab 发展战略：元反思诊断

> 2026-04-18 | 基于四层元架构框架 + 《Thin Harness, Fat Skills》对照分析

## 诊断结论

项目已完成 35 个 REQ，治理框架的"骨架"已经立住了。但对照《Thin Harness, Fat Skills》的四层架构和元反思框架审视，**当前最大的结构性缺口不是功能不足，而是"经验不会回流"**。

---

## 四层架构诊断

### Layer 1 认知边界层 — 评分：70/100

| 状态 | 项 |
|------|-----|
| ✅ | 4 实体约束、REQ 颗粒度检查、会话启动恢复 |
| ❌ | **Skill 参数化**：REQ 模板是一次性文档，不是可复用过程 |
| ❌ | **问题分类**：所有 REQ 走同一个模板，没有 bugfix/feature/refactor 区分 |

### Layer 2 意图防护层 — 评分：75/100

| 状态 | 项 |
|------|-----|
| ✅ | PreToolUse 强制阻断、豁免审计、错误分类器 (E001-E008) |
| ❌ | **人类侧 DX**：安装后 6 步手动配置，认知悬崖陡峭 |
| ❌ | **项目健康检查**：没有 `harness-doctor` 类的自检能力 |

### Layer 3 执行可靠层 — 评分：65/100

| 状态 | 项 |
|------|-----|
| ✅ | REQ 生命周期 CLI、循环检测、SessionStart 恢复 |
| ❌ | **经验不回流**：35 篇 experience 文档写完就躺平，不反哺操作 |
| ❌ | **Stop/SessionEnd/PreCompact** hook 全部缺失（路线图 Phase 1 才补） |
| ❌ | 单 REQ 串行：不支持并发 REQ 依赖关系 |

### Layer 4 持续优化层 — 评分：40/100

| 状态 | 项 |
|------|-----|
| ✅ | Experience 沉淀机制存在 |
| ❌ | **无学习闭环**：经验 → 模式 → 注入 skill 这条链路完全断开 |
| ❌ | **无自学习**：skill 不会自我改写 |
| ❌ | **无上下文路由**：没有 resolver 按需加载 |

## 核心洞察：经验断裂

```
当前状态（单向）：
REQ 实施 → 经验写入 context/experience/ → 💀 终止（不再被消费）

目标状态（闭环）：
REQ 实施 → 经验写入 → 模式提取 → 自动注入到 skill/hook → 下一个 REQ 受益
              ↑                                           |
              └─────────────── 验证效果 ←──────────────────┘
```

《Thin Harness, Fat Skills》的核心观点：

> *"The skill rewrites itself. The next run uses them automatically."*

harness-lab 的 experience 文档做到了"记录"，但没做到"回流"。第 35 个 REQ 和第 1 个 REQ 面对的治理规则是一样的——系统没有从自己的历史中学到任何东西。

---

## 发展方向建议（按优先级排序）

### 方向 1：学习闭环 — 经验回流机制

**为什么排第一**：这是 10x 杠杆点。当前 35 篇 experience 是沉睡资产，激活它们比写任何新功能都更有价值。

**具体做什么**：
- 建一个不变量提取器，扫描 `context/experience/` 中的失败模式
- 提取为结构化规则（如 `INV-001: 模板占位符逃逸`）
- 在 PreToolUse hook 中按文件路径匹配，自动注入相关不变量警告
- 每次新 experience 写入后，触发重新扫描

**不与路线图冲突**：这等于把路线图 Phase 4.2（不变量提取器）前置，因为它不依赖 Phase 1-3 的任何能力。

### 方向 2：Onboarding DX — 降低接入摩擦

**为什么重要**：当前安装后 6 步手动配置是认知悬崖。一个新用户在接入的前 30 分钟内很可能放弃。

**具体做什么**：
- `harness-doctor` 命令：一键诊断项目健康度（hook 是否生效、REQ 模板是否已定制、experience 目录是否有内容）
- 交互式首个 REQ 向导：接入后引导创建第一个真实 REQ，而非留空模板
- 适配器模式：提供 React/Python/Go 等常见项目类型的预设配置

### 方向 3：Skill 参数化 — 工作流特化

**为什么重要**：《Thin Harness, Fat Skills》的核心洞察是 skill = method call。当前 REQ 模板是"通用型"，但 bugfix 和新 feature 的流程差异很大。

**具体做什么**：
- `/req:bugfix` — 默认带复现步骤、影响范围、回归测试
- `/req:feature` — 默认带用户故事、设计稿、验收标准
- `/req:refactor` — 默认带重构目标、依赖分析、迁移计划
- 每个"特化模板"是通用模板的子集 + 特定字段

### 方向 4：按路线图推进 Phase 1（Stop 评估器）

这是已有路线图中最紧迫的一步——**防假完成**。AI 声称完成但实际没完成是自治场景下最大的风险。

### 方向 5（远期）：跨项目模式共享

多个使用 harness-lab 的项目之间共享脱敏后的不变量模式。这是《Thin Harness, Fat Skills》里"skills compound"的终极形态，但需要先解决前 4 个方向。

---

## 建议的下一步

最小行动路径：

1. **创建一个 REQ**：从"方向 1（学习闭环）"切入，先做最小版本——只提取 experience 中的错误模式并注入 PreToolUse 提醒。预计 2-3 个文件改动。
2. 这一步完成后再做"方向 2（harness-doctor）"，同样是小范围改动。
3. 两个方向互不依赖，可以交替推进。

## 与现有路线图的关系

本方案与 `docs/plans/autonomous-mode-roadmap.md` 的关系：

| 路线图 Phase | 本方案对应 | 差异 |
|-------------|-----------|------|
| Phase 1（防假完成） | 方向 4 | 一致，优先级下调 |
| Phase 4.2（不变量提取器） | 方向 1 | **前置**，不依赖 Phase 1-3 |
| — | 方向 2（DX） | 新增，路线图未覆盖 |
| — | 方向 3（参数化） | 新增，路线图未覆盖 |

核心差异：本方案在路线图之外插入了 DX 和学习闭环两个独立方向，并将不变量提取器从 Phase 4 前置到第一阶段。
