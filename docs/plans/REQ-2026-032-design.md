# REQ-2026-032 设计文档：反馈机制增强

## 概述

本设计文档描述如何将"元反思"和"对齐"机制整合到 REQ 流程的反馈环节中。

## 核心概念

### 元反思（Meta-Reflection）

**定义**: 对实施过程和结果的反思，检查是否真正实现了目标，是否存在隐含问题。

**时机**: verify 阶段（实施完成后，进入 fix 之前）

**检查清单**:
1. 目标实现: 最初的目标是否全部达成？有无遗漏？
2. 旧功能保护: 是否破坏了现有功能？
3. 逻辑正确性: 实现中是否存在重大逻辑错误？
4. 完整性: 是否有未处理的边界情况？
5. 可维护性: 代码/文档是否易于理解和维护？

### 对齐（Alignment）

**定义**: 对照最初的计划和规范，检查实现是否与之保持一致。

**时机**: record 阶段（修复完成后，记录经验之前）

**检查维度**:
1. 目标对齐: 实现是否服务于最初的目标？
2. 设计对齐: 实现是否符合设计文档？
3. 验收标准对齐: 所有验收标准是否都有对应实现？

## 流程整合

```
plan -> build -> verify -> fix -> record
              ↓           ↓
         [元反思]    [对齐检查]
```

## 实现方案

### 1. REQ 模板更新

在 `requirements/REQ_TEMPLATE.md` 中:
- verify 阶段增加"元反思检查清单"小节
- record 阶段增加"对齐检查"小节

### 2. CLAUDE.md 更新

更新反馈环节描述，明确标注:
- verify 阶段必须执行元反思
- record 阶段必须执行对齐检查

### 3. 可选脚本工具

创建 `scripts/req-reflect.mjs`:
- 读取 REQ 文件
- 输出元反思问题清单
- 支持交互式填写或生成报告模板

## 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 增加认知负担 | 检查清单控制在 5 项以内 |
| 执行者敷衍 | 要求输出结构化报告 |
| 与现有流程冲突 | 作为可选增强，不强制阻断 |

## 验证方式

1. 使用本 REQ 自身验证流程
2. 选择下一个 REQ 进行试点
3. 收集团队反馈，迭代优化
