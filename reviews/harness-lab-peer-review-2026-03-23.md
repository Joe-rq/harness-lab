## Round 1：事实层（只写事实，不评价）

### PM — 产品视角

**产品定位：**
- 定位为"研发治理层模板"，非业务运行时框架（AGENTS.md:3-5）
- 目标：让人和 agent 按同一套入口接手工作，依赖仓库内证据链而非口头上下文（README.md:10-14）
- 内嵌到已有仓库，不强行替换目标项目的目录结构、架构或运行时（README.md:21-23）
- 版本 1.1.0（package.json:3）

**功能完成度：**
- REQ 生命周期：95%（有 in-progress/completed 流程，搁置流程文档化但无专用目录）
- 设计评审（CEO/Design/Eng 三层）：100%
- 代码评审：100%（有约束要求真实命令执行和发现追溯）
- QA：100%（覆盖验收/回归/环境验证）
- 发布：100%（覆盖发布前检查/Smoke/回滚）
- 上下文恢复：90%（业务 context 只有索引无存根；技术 context 有 5 个存根 doc）
- 经验沉淀：80%（有框架和示例，尚无真实项目贡献的经验 doc）
- 跨会话交接：100%（.claude/progress.txt 完全实现）
- 验证命令绑定：100%（防守脚本+接入指导到位）
- 所有 6 个 skill 都有完整内容（输入预检查、执行要点、报告模板和约束）
- 完整示例链路：REQ-2026-900 展示从设计到报告的全链路

**竞品功能对比：**
- vs Shape Up：Harness Lab 无时间盒迭代机制
- vs DORA：Harness Lab 提供流程框架，不提供指标体系
- vs ADR：context/plans 中记录设计决策，但不如 ADR 那样强制独立 decision 文档
- vs RFC：skills/plan 中的三层评审类似 RFC，但更内化更强制
- Harness Lab 的特异性：显式支持人-AI 协作、跨会话上下文保留、强制仓库内证据链、经验沉淀机制

**目标用户线索：**
- 同时面向人类开发者和 AI agent（AGENTS.md:5, README.md:10-14）
- CLAUDE.md 既指导 AI agent，也对人提出工作约束
- README.md 有 5 步快速开始指南（第 53-108 行）
- 不硬依赖 Claude Code，但文件名和示例面向 Claude
- 适用场景：多人持续迭代的产品仓库、人-AI 混合协作
- 不适合：一次性脚本、半天 demo、无需知识沉淀的实验仓库

### Engineer — 技术视角

**文件结构事实：**
- 41 个文件，最大目录深度 2 层
- 最大单文件 292 行（reviews/harness-lab-governance-review.md，接近 300 行警戒线但为特殊文件）
- 所有目录文件数 ≤ 9，基本满足"每层不超过 8 个"的约定
- 命名约定一致性 95%：顶层大写（AGENTS.md），子目录内小写连字符（tech-stack.md），REQ 文件统一前缀
- 所有含内容目录都有 README.md（requirements/ 用 INDEX.md 替代）
- .gitignore 包含对 context/session/*.md 的忽略规则，但 context/session/ 目录不存在

**模板与约定设计：**
- REQ 编号：REQ-YYYY-NNN-<title>.md（4 位年份 + 3 位序号 + 标题）
- 生命周期双维追踪：状态维度（draft/in-progress/blocked/completed）× 阶段维度（design/implementation/review/qa/ship）
- 报告 3 类（code-review/qa/ship），最低内容 8 项要求（关联 REQ/日期/执行人/输入/命令/结果/是否阻塞/后续）
- context/ 分类边界清晰：business=为什么做，tech=怎么做，experience=做过什么
- 经验文档命名：YYYY-MM-DD-<topic>.md 或 best-practice-<topic>.md

**自动化与工具集成：**
- package.json 的 lint/test/build/verify 全部为防守脚本（主动失败，迫使接入方绑定真实命令）
- .claude/settings.local.json 声明了 verify-governance.sh 的执行权限，但该脚本不存在
- 无 GitHub Actions 配置（.github/workflows/ 不存在）
- 无 Git hooks 自定义
- 有 GitHub PR/Issue 模板（.github/PULL_REQUEST_TEMPLATE.md, .github/ISSUE_TEMPLATE/）

**一致性与耦合：**
- AGENTS.md vs CLAUDE.md：补充而非冲突——AGENTS 定义治理规则，CLAUDE 定义 Agent 行为约束
- 读取顺序三处定义 100% 一致
- 所有路径引用 100% 有效
- 零循环引用（所有链接单向：REQ → design → reports → experience）
- 重叠内容互补：AGENTS 说"是什么"，CLAUDE 说"何时算完成"

**扩展性事实：**
- 非侵入式设计：保留目标项目自身业务结构，仅在外层套治理协议（AGENTS.md:102-103）
- 显式依赖仅 Git + Markdown 编辑器，Claude Code 可选但非必需
- 非 Claude Code 用户可用性：高（所有核心文件为 Markdown，无 Claude 专用语法）
- 不可适配的约束：REQ 编号格式、报告文件位置、生命周期状态名、读取顺序

### 独到发现

1. **[PM 交叉发现] 验证命令绑定"100%完成"与实际脱节**：PM 自评为 100%，但 settings.local.json 引用不存在的脚本、无 CI 自动验证——表面完成而非实质完成。
2. **[Designer 交叉发现] 入口导航歧义**：三处定义读取顺序（AGENTS.md/CLAUDE.md/README.md），但新用户打开仓库时不知道先读哪个。建议明确首个入口。
3. **[Designer 交叉发现] 人-AI 协作的文档双轨制未显式化**：AI agent 自动恢复上下文，人类用户需手动阅读。文档对两类用户没有区分说明。
4. **[Engineer 交叉发现] 示例文件混在工作目录中**：REQ-2026-900 示例分散在 completed/、plans/、reports/ 中，接入方可能误将其当成真实工作文件。
5. **[Engineer 交叉发现] context/ 缺少演进/弃用策略**：经验 doc 会随时间过时，但无机制标记 deprecated 或清理。
6. **[PM 交叉发现] 目标用户线索模糊**：skills/ 面向 AI agent（Claude Code context），README 面向人类，优先级不清。

## Round 3：共识层

### 🚨 关键质询 3 回应：防守脚本悖论

**裁决：这是有意的设计取舍，但执行上有两个实际缺陷需修复。**

**论证：**
- 模板仓库是"治理协议层"，不是运行中的业务项目。期望它有真实的 lint/test/build 命令，如同期望建筑蓝图有可用的供水系统——逻辑层级不对。
- 防守脚本的设计意图正确：迫使接入方绑定真实命令，防止误判模板为已验证项目。
- **但 Critic 正确指出了两个实际缺陷：**
  1. settings.local.json 引用不存在的 verify-governance.sh — 这是配置污染，必须修复
  2. 示例报告应更显式地标注"结构参考，非真实执行记录" — 当前标注不够醒目

**行动：**
- P0：清理 settings.local.json 无效引用（或补全脚本）
- P1：在 README 中增加"模板仓库的自验证边界"说明，明确模板不含真实命令

## 被低估的优势

1. **四阶段 Skills 框架的完整闭环**：plan（3 层评审）→ review → qa → ship，覆盖需求到发布全链路，且每个 skill 都有输入/执行/输出/约束的完整定义。这比绝大多数开源治理模板完善。
2. **验证计划作为 REQ 必填项**：REQ_TEMPLATE.md 强制要求填写验证计划，从源头防止"纸面设计未验证就发布"。
3. **防守脚本的安全网设计**：package.json 的失败脚本看似激进，实际上是唯一能防止"接入方忘记绑定真实命令"的机制。
4. **证据链思维**：报告不是附属文档，而是 REQ 的组成部分（experience/2026-03-20-example-evidence-chain.md 明确说明），这种设计哲学在同类工具中罕见。
