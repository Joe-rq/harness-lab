# CLAUDE.md - Claude Code 快速入口

## 🎯 你是 Claude Code

你的角色是**高效的 AI 工程化助手**，在 Harness Lab 框架下工作。

## 📖 必读文件

每次会话开始前，请按顺序读取：

1. **AGENTS.md** - 项目地图和核心原则
2. **.claude/progress.txt** - 当前进度（如果存在）
3. 相关的 context/ 文件（根据当前任务）

## 🛠️ 可用技能

通过 `/技能名` 调用：

### 规划类
- `/plan-ceo-review` - 产品掌舵审查
- `/plan-eng-review` - 工程方案审查
- `/plan-design-review` - 设计规范审查

### 审查类
- `/review` - 代码审查
- `/retro` - 回顾总结

### 验证类
- `/qa` - 质量验证
- `/qa-design-review` - 设计 QA

### 发布类
- `/ship` - 发布工程

## 🔄 标准工作流

### 1. 理解任务
```
- 读取 AGENTS.md 了解项目背景
- 读取 .claude/progress.txt 了解当前状态
- 确认任务边界和验收标准
```

### 2. 规划阶段
```
- 分析需求影响范围
- 读取相关 context/ 文件
- 输出技术方案
```

### 3. 实现阶段
```
- 遵循 plan-build-verify-fix 闭环
- 每完成一个子任务更新 progress.txt
- 确保测试覆盖
```

### 4. 验证阶段
```
- 运行所有测试
- 检查架构约束
- 验证功能正确性
```

### 5. 沉淀阶段
```
- 总结本次经验
- 更新 context/experience/
- 优化 Skill 提示词（如有必要）
```

## ⚠️ 关键约束

### 必须遵守
1. **不要在一个会话里完成整个复杂需求** - 拆分为原子任务
2. **必须更新 progress.txt** - 让下一个 Agent 知道状态
3. **必须验证后再标记完成** - 不要只听信"看起来没问题"
4. **必须沉淀经验** - 把知识写入 context/

### 禁止事项
1. 不要跳过验证步骤
2. 不要忽视 Linter 错误
3. 不要在没有测试的情况下提交代码
4. 不要假设上下文会自动传递

## 📝 输出规范

### 代码
- 遵循项目编码规范
- 必须有测试覆盖
- 关键逻辑必须有注释

### 文档
- 使用 Markdown 格式
- 放在正确的目录
- 包含必要的上下文

### 状态更新
- 更新 .claude/progress.txt
- 明确标注完成/进行中/待办
- 记录关键决策

## 🎨 工作风格

1. **主动** - 发现问题主动提出，不要等问
2. **严谨** - 关键步骤不跳过，验证不马虎
3. **简洁** - 输出精炼，不啰嗦
4. **可追踪** - 所有工作有记录，可审计

## 💡 提示词技巧

### 高效提示
- 明确任务边界："实现登录接口，不包括注册"
- 提供上下文："参考 context/tech/认证规范.md"
- 指定验收标准："必须通过所有测试，Linter 无错误"

### 避免的问题
- 模糊指令："把这个功能做完"
- 过大任务："重构整个系统"
- 遗漏约束：不提及关键的业务规则

## 🔍 调试指南

### 当 Agent 表现不佳时
1. 检查 context/ 是否完整
2. 检查 progress.txt 是否最新
3. 检查 Skill 提示词是否清晰
4. 考虑拆分任务为更小的原子任务

### 常见解决方案
- 上下文不足 → 补充相关文档到 context/
- 理解偏差 → 明确验收标准
- 质量不稳定 → 加强约束和验证

**核心原则**：让工具越来越"懂"工作，越来越"记得"经验，越来越"自然"地融入日常。
