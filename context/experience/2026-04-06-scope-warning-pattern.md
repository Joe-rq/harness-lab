# 2026-04-06 4 实体规则的轻量级实现（深度思考案例）

## 场景

用户提出："约束实体（4 实体规则）有必要添加进来，REQ 的颗粒度需要定义好"

初始理解的问题：
- 如何定义"实体"？（文件？模块？API 端点？）
- 阈值是 4 还是 5？如何论证？
- 如何检测？谁来执行？
- 超限时如何拆分？策略是什么？

看起来需要一个完整的 REQ 来系统化设计。

## 关联材料

- 阳志平文章：`yangzhiping.com.md`（技巧 2：约束实体）
- 修改的 hook：`scripts/req-check.sh`
- 更新的文档：`CLAUDE.md`（4 实体规则说明）

## 深度思考过程

### 问题转换

**原始问题**：定义 REQ 颗粒度标准，将 4 实体规则纳入体系

**重新定义后的问题**：如何设计一个低认知负担的 REQ 拆分预警机制？

关键洞察：
- "计数实体"本身就有认知负担——要数清楚涉及哪些文件/模块，需要先理解整个改动范围
- 规则的价值在于"触发反思"，而非"精确执行"
- 当系统说"你改了 4 个文件"时，人会本能地想"是不是太多了？"这个反思本身就是价值

### 业界解法映射

| 领域 | 模式 | 映射到本问题 |
|------|------|-------------|
| IDE 代码检查 | 复杂度超标时标黄线 | 文件数超限时弹出提示 |
| PR 平台 | >400 行难 review | 修改文件数作为理解难度代理 |
| 负载均衡 | 健康检查 + 动态调整 | 监控完成时间，调整策略 |

最优转换：把"颗粒度定义"→转换为"认知负载预警"

### 轻量级方案

不是"定义标准"，而是：
1. **简单规则**：涉及 4+ 文件时提示
2. **技术实现**：PreToolUse hook 检测修改文件数
3. **人的角色**：确认是否真的需要这么多文件，而不是机械拆分

## 关键决策

- **决策 1：不创建 REQ-2026-033**
  这个问题不需要完整 REQ，只需要一个 hook 增强 + 文档一句话

- **决策 2：预警而非阻断**
  不是"超过 4 个就不允许"，而是"超过 4 个就问你一下"
  保留人的判断权，只增加一个反思触发点

- **决策 3：模糊计数**
  不精确定义"实体"是什么，用 git 追踪的修改文件数作为代理指标
  排除 requirements/ 和 docs/（这些是 REQ 内容本身）

- **决策 4：直接修改而非系统化设计**
  修改现有 `req-check.sh`，增加 10 行代码
  更新 CLAUDE.md 增加一段话
  当天完成，当天生效

## 解决方案

```bash
# req-check.sh 新增逻辑
MODIFIED_COUNT=$(git status --porcelain=v1 | grep -E '^\s*[MADRC]' | grep -vE 'requirements/|docs/' | wc -l)

if [ "$MODIFIED_COUNT" -ge 4 ]; then
  echo "⚠️  Modified $MODIFIED_COUNT files. Consider splitting with TaskCreate."
fi
```

## 复用建议

- **问题过于复杂时**：用深度思考（sequential-thinking MCP）重新定义问题
  - 原始问题可能包含了不必要的假设
  - 转换后的版本往往更简单、更可执行

- **"系统化" vs "轻量"的权衡**：
  - 如果规则需要详细定义边界条件 → 系统化 REQ
  - 如果规则是启发式、触发式的 → 轻量 hook + 文档

- **4 实体规则的应用边界**：
  - 适用于：感知复杂度、触发反思
  - 不适用于：精确容量规划、强制限制

- **技巧 3（重新定义问题）的实际应用**：
  不要解决"如何精确定义颗粒度"这个复杂问题
  而是解决"如何在临界点触发反思"这个简单问题

## 一句话总结

> 规则的价值不在于精确，而在于让人记得要问一问。
