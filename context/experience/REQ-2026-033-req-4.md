# 2026-04-02 REQ 颗粒度 4 实体原则

## 场景

现有治理框架缺少"一个 REQ 应该多大"的明确上限。CLAUDE.md 只规定"3+ 文件需要 REQ"，但没规定一个 REQ 最多涉及多少实体。这可能导致 4 目标但 8 文件的 REQ，违反"最佳对话模式 = 4 实体以内"原则。

## 关联材料

- REQ: `requirements/in-progress/REQ-2026-033-req-granularity-4-entity-rule.md`
- Code Review: `requirements/reports/REQ-2026-033-code-review.md`
- QA: `requirements/reports/REQ-2026-033-qa.md`

## 问题 / 模式

- **规则不完整**：有"要不要 REQ"的判断，没有"一个 REQ 多大"的上限
- **实践盲区**：新项目引入时缺乏颗粒度判断依据
- **原则缺口**：对话效率原则（4 实体）未映射到 REQ 设计

## 关键决策

- **决策 1**：采用"4 实体原则"作为 REQ 颗粒度上限
- **决策 2**：实体定义 = 文件 + 模块 + 概念
- **决策 3**：自检清单 5 项，任何一项不通过就考虑拆分
- **决策 4**：不在 CLI 自动检测（保持轻量，未来可扩展）

## 解决方案

1. 在 CLAUDE.md 添加"检查 2: REQ 颗粒度是否合适？"
2. 更新判断表：3-4 文件 → 1 个 REQ，5-8 文件 → 2 个 REQ，9+ 文件 → 必须拆分
3. 在 REQ_TEMPLATE.md 添加颗粒度自检清单章节
4. 在 README.md 说明颗粒度原则

## 复用建议

- **新项目引入**：先用判断表确定需要几个 REQ，再按功能边界拆分
- **颗粒度争议**：用自检清单作为判断标准
- **规则扩展**：未来可在 `req:create` 时自动提示颗粒度建议
