# 经验 Context 索引

> 本目录用于沉淀可复用的经验，而不是堆放零散笔记。

## 什么时候写经验文档

建议在这些场景落盘：
- 一个 REQ 完成后，出现了可复用的工程模式
- 踩到了会重复出现的坑
- 修稳了验证链路或发布链路
- 发现某个 Skill 或工作流需要长期更新

## 与 REQ 的关系

经验文档应尽量链接回：
- 对应的 REQ
- 相关设计稿
- 相关 review / QA / ship 报告

## 命名建议

- `YYYY-MM-DD-<topic>.md`
- `best-practice-<topic>.md`
- `pitfall-<topic>.md`
- `postmortem-YYYY-MM-DD-<incident>.md`

## 最低内容

每份经验文档至少说明：
- 场景
- 问题或模式
- 根因或关键判断
- 解决方案
- 后续项目如何复用

## 原则

- 没有复用价值的聊天摘要不要放进来
- 经验要可执行，能直接影响下次工作方式
- 同一类经验优先补到已有文档，而不是重复创建近似文件
