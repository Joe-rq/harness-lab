# REQ-2026-032: 反馈机制增强：元反思与对齐

## 状态
- 当前状态：completed
- 当前阶段：qa

## 背景

### 问题
当前 REQ 流程的反馈环节（verify → fix → record）存在结构缺陷：

1. **缺乏元反思机制**：实施完成后，没有强制步骤回顾"是否实现了目标？是否破坏了旧功能？是否存在重大逻辑错误？"
2. **缺乏对齐检查**：没有结构化步骤对照"最初计划 vs 实际实现"，导致计划与实现脱节
3. **反馈流于形式**：verify 阶段往往只是"跑一遍测试"，没有深度检验

### 启发来源
阳志平《让 AI 变得更聪明的 12 个元反思技巧》中的"技巧 1: 元反思"和"技巧 9: 对齐"指出：
- **元反思**是 AI 产出的关键——强制反思必然发现新问题
- **对齐**=利用大模型的补全能力，不完善的地方互相补全
- 这两个技巧对于复杂任务的质量保障至关重要

### 目标价值
将元反思和对齐固化为 REQ 反馈环节的标准步骤，**系统性提升 REQ 完成质量**，减少"看似完成、实则遗漏"的情况。

## 目标

- [ ] **目标 1**: 在 REQ 流程中增加"元反思检查点"（verify 阶段）
  - 定义元反思的触发时机、检查清单、输出格式
  - 提供脚本/工具支持（如 `npm run req:reflect`）

- [ ] **目标 2**: 在 REQ 流程中增加"对齐检查点"（fix/record 阶段）
  - 定义对齐的检查维度（目标对齐、设计对齐、验收标准对齐）
  - 提供结构化的对齐报告模板

- [ ] **目标 3**: 更新 REQ 模板和相关文档
  - 修改 `requirements/REQ_TEMPLATE.md`，在 verify/record 阶段嵌入元反思和对齐要求
  - 更新 `CLAUDE.md` 中的流程说明

- [ ] **目标 4**: 验证机制有效性
  - 使用本 REQ 自身验证新流程
  - 至少完成一个其他 REQ 的试点应用

## 风险与回滚
- **风险 1**: 新增的反馈步骤增加认知负担，导致执行者敷衍了事
  - **缓解**: 保持检查清单精简（每个环节 ≤5 项），提供明确的输出模板

- **风险 2**: 元反思和对齐的界限模糊，执行者困惑
  - **缓解**: 在模板中明确区分两者的关注点和时机

- **回滚方式**: 
  - 撤销对 REQ_TEMPLATE.md 和 CLAUDE.md 的修改
  - 删除新增的 npm scripts
